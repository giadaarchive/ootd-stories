# L's Lookbook Style — OOTD Skills

This file documents how the OOTD database works: what it records, how to create and name entries, how the story generation system works, what the editorial style rules are, and how the Substack publishing pipeline runs.

**Notion database:** `235ccd15cda18097be05ec7a19f9f39a`

---

## What this database is

One row per outfit worn. It is three things in one:

1. **Wear log** — a date-stamped record of what was worn, linking back to specific items in the wardrobe database. This is what drives cost-per-wear calculations.
2. **Editorial archive** — each entry has an OOTD Story: an atmospheric, scene-setting narrative written by Claude from the outfit photos. Not a description of the clothes. A story about the world the outfit belongs to.
3. **Publishing pipeline** — entries marked `Post to Substack` are automatically picked up by `substack.py` and published to the Substack newsletter.

---

## Properties

| Property | Type | Notes |
|----------|------|-------|
| `Name` | title | Entry label — usually the date, sometimes descriptive (see naming below) |
| `Worn` | date | The date the outfit was worn — the primary sorting field |
| `Added` | created_time | When the row was created in Notion |
| `Items` | relation | Links to specific garments in L's Collection of Amazing Pieces |
| `OOTD Story` | rich_text | The editorial narrative — generated by `lookbook.py` or written manually |
| `Substack` | status | `No Post` / `Post to Substack` / `Posted` — controls the publishing pipeline |
| `Favourite` | checkbox | Mark entries worth revisiting or highlighting |
| `Colours` | rollup | Colour tags of all linked items — pulled from the wardrobe database |
| `Cost` | rollup | Total actual purchase cost of all linked items in SGD |
| `Retail` | rollup | Total retail price of all linked items in USD |

## Naming conventions

The `Name` field defaults to the date (e.g. `2026-04-19`) but can be overridden with something descriptive when the entry has a clear context.

| Format | When to use | Examples |
|--------|-------------|---------|
| `YYYY-MM-DD` | Standard daily outfit | `2026-04-19` |
| Descriptive noun phrase | Outfit has a distinct occasion or character | `bread run`, `lounge and PJ`, `airport` |
| `@TodayTheme` (no space) | Multiple looks from the same styled session with a shared theme | `@TodayDark Academia` |

The `@Today` prefix signals that this is one of several looks created in a single session — a styled editorial set rather than a single day's outfit.

Multiple entries can share the same `Worn` date. This happens when there is more than one outfit worn that day, or when a session produces multiple distinct looks.

## Creating an entry

1. Create a new row. Set `Worn` to the date.
2. Name it — date by default, descriptive if the occasion warrants it.
3. Upload outfit photos to the page body (drag directly into the page).
4. Link `Items` — connect every garment worn in this outfit to its row in the wardrobe database. This is what feeds the wear count and CPW calculations.
5. Run `lookbook.py` — it finds all entries without a story, reads the images, and writes the `OOTD Story` field automatically.
6. Review the story. If posting to Substack, set the status to `Post to Substack`.

## The story generation system

`lookbook.py` runs against the database, finds every entry without an `OOTD Story`, reads up to 3 images from the page body, sends them to Claude with the editorial prompt, and writes the result back to the `OOTD Story` field.

**To run:**
```bash
python3 lookbook.py
```

It processes all entries without stories, in order from most recently added. Rate-limited to avoid hitting Notion or Anthropic limits.

### What drives the story

The script reads the outfit photos visually. It does not read the item names or descriptions — the images are the input. This means the story reflects what the outfit actually looks like, not what the database says it is.

### Story style rules

The editorial voice is consistent across all entries. These rules are baked into the prompt and should be maintained if the prompt is ever edited:

**Tone and register:**
- Chic, atmospheric, scene-setting — Vogue editorial caption meets personal essay
- British English throughout
- Written by someone who knows both fashion and literature
- The philosophy: every day is an occasion. The outfit is how someone chose to show up. Make that visible.

**Structure:**
- 3–4 paragraphs
- Never describe the clothes literally. Write about the feeling, the world, the character, the cultural moment.
- Begin in scene — not with "I", not with a direct description of what is being worn
- Give the look a title when it has one (bold or heading)

**Sentence-level rules:**
- No contradiction structures: do not write "She is not X, Y, or Z. She is W." Build meaning through what something IS.
- No em dashes. If an em dash feels necessary, restructure the sentence instead.
- Vary sentence structures across entries — avoid falling into a signature opening that repeats every time

**Format rotation:**
Stories should vary in their entry point and structure across different entries. Formats to rotate through:
- **Vibe/atmosphere** — opens with a setting or mood ("Somewhere between midnight and meaning...")
- **Character study** — opens with a person or type ("There is a particular kind of woman who...")
- **Garment focus** — one specific piece is the anchor and the story builds from it
- **Second person** — addresses "you" directly, placing the reader in the look
- **Event/occasion** — the occasion or destination shapes the whole narrative

Never use the same opening construction in consecutive entries. Never repeat a title across entries.

**Things to avoid:**
- "She is not...": contradiction structures (see above)
- Mentioning mirrors, getting dressed, looking at oneself
- Literal inventory of what is worn ("the cream trousers with the silk blouse and the...")
- Trend references that will date the piece
- Aspirational boilerplate ("the kind of woman who...")

## The Substack pipeline

Three status values control publication:

| Status | Meaning |
|--------|---------|
| `No Post` | Default. Will never be published. Used for private looks, loungewear, styled sessions not intended for the newsletter. |
| `Post to Substack` | Queued for publishing. `substack.py` picks this up on the next run and publishes it. |
| `Posted` | Published. Set automatically by `substack.py` after successful posting. |

**To queue a post:** set the Substack status to `Post to Substack`. The script picks it up on the next run.

**The script:** `substack.py` — reads all entries with `Post to Substack` status, publishes them to `giadaarchive.substack.com`, and marks them as `Posted`.

Not every entry needs to be published. Most daily outfits stay at `No Post`. Post to Substack when the look and the story are both strong enough to stand as a public editorial piece.

## The rollup properties

**`Colours`** — pulls colour tags from all linked items. Useful for reviewing the palette of a look — whether the colour story is coherent, whether there is a dominant hue.

**`Cost`** — sum of the actual purchase prices (SGD) of all linked items. Shows the real cost of the outfit as assembled.

**`Retail`** — sum of retail prices (USD) of all linked items. Shows what the outfit would have cost at full retail, and by comparison with `Cost`, how much was saved through vintage and pre-owned sourcing.

## How to read the database

- **By date:** sort by `Worn` descending — most recent first
- **Best looks:** filter where `Favourite` is checked
- **Ready to publish:** filter where Substack = `Post to Substack`
- **Published archive:** filter where Substack = `Posted`
- **Styled sessions:** filter by `Name` contains `@Today` to see all editorial multi-look entries
- **By item:** open any wardrobe item in L's Collection of Amazing Pieces and look at the `How L Styles` relation — it shows every OOTD entry where that piece was worn
